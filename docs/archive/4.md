# Architectural Blueprint for a Developer-Grade Local AI Command Center on Next.js 15

## System Architecture and Runtime Foundation

The development of a native "Command Center" for local AI models necessitates a carefully architected stack built entirely on on-premise resources, excluding any form of cloud inference [[42]]. The foundation of this architecture rests upon the choice of a local Large Language Model (LLM) runtime, which serves as the engine for all subsequent functionalities, from basic chat interactions to complex agent workflows. This section provides a comprehensive analysis of the available local runtimes, culminating in a detailed comparison and a justified recommendation for Ollama as the primary backend. It further outlines the complete system architecture, presenting a Mermaid flowchart that details the data flow between the Next.js frontend, the API layer, the model runtime, the vector store, the persistence layer, and the hardware monitoring subsystem. Each component is defined by its role and the protocols used for communication, establishing a clear blueprint for the entire application.

The selection of a local LLM runtime is a pivotal decision that profoundly influences the project's scope, performance, and development trajectory. The primary contenders in the 2025 landscape include Ollama, llama.cpp's native HTTP server, LocalAI, and LM Studio's server mode [[26,40,41]]. While Ollama has emerged as a de facto standard due to its ease of use and standardized REST API, a thorough evaluation of its alternatives is essential to ensure it is the optimal choice for a developer-grade command center [[27,34]]. The key criteria for comparison are OpenAI API compatibility, streaming protocol support, multimodal capabilities, GPU utilization efficiency, friction for Node.js integration, active maintenance status projected through early 2026, and the ideal use case for each platform [[41]].

Ollama distinguishes itself through its user-friendly CLI for managing models and its straightforward REST API, which simplifies the process of pulling, running, and interacting with a wide variety of open-source models [[34]]. Its API endpoints for generating responses, managing chats, and listing models provide a consistent interface for developers [[70]]. Streaming is handled via Server-Sent Events (SSE), a standard web technology well-suited for pushing token-by-token updates from the server to the client, making it a natural fit for a live chat interface [[31]]. However, while Ollama supports multimodal models, the patterns for inputting images or other non-textual data may not be as mature or flexible as those offered by more specialized servers [[73]]. Its reliance on the underlying `llama.cpp` engine means it inherits some of its performance characteristics but also abstracts away direct control over fine-grained parameters [[40]].

In contrast, the native HTTP server provided by `llama.cpp` offers a path to potentially superior performance and greater control. As `llama.cpp` is becoming the "source of truth" for local AI inference, many popular applications, including Ollama itself, rely on it, indicating its stability and performance merits [[40]]. A benchmark report suggested that `llama.cpp`'s integration with Ollama could yield a 1.5x faster inference rate (311 vs. 204 tokens/sec) and enables dynamic model discovery with zero overhead [[24]]. This raw performance advantage makes it a compelling option for compute-intensive tasks. It supports streaming and function calling, positioning it as a powerful alternative for performance-critical applications [[24]]. The primary trade-off is a steeper learning curve; integrating directly with `llama.cpp` requires a deeper understanding of its C++ codebase and compilation flags compared to Ollama's simple `ollama run <model>` command. For a command center prioritizing maximum performance and low-level control, this presents a viable, albeit more complex, path forward.

LocalAI is another significant competitor to Ollama, also designed with enterprise deployments in mind [[41]]. It aims for OpenAI API compatibility and emphasizes security features that may be advantageous for team-based or production environments within an organization [[41]]. Both platforms excel at keeping data on-premise, but their security characteristics differ, making LocalAI a strong candidate for deployments with stringent security requirements [[41]]. Its maintenance status appears healthy, positioning it as a reliable long-term option. LM Studio, while often perceived as a desktop GUI-first application, also offers a server mode that exposes its models over a network interface [[26]]. This mode is known for being developer-friendly, which could reduce initial setup friction for engineers already familiar with the LM Studio ecosystem [[26]]. Its main strength lies in its user-centric design, which prioritizes accessibility and ease of use over raw performance or configurability.

Based on this analysis, **Ollama is the recommended primary runtime** for the Command Center. Its combination of a standardized REST API, active maintenance, a dedicated JavaScript library, and a vast community makes it the most pragmatic choice for rapid development and long-term viability [[34,70]]. The slight abstraction over raw performance offered by `llama.cpp` is a reasonable trade-off for the significant reduction in development friction and time-to-market. Should future performance bottlenecks emerge, a strategic refactoring to integrate directly with `llama.cpp`'s HTTP server remains a viable optimization path [[24,40]].

## Core Application Architecture in Next.js 15

The architectural integrity of the Command Center hinges on the effective implementation of the Next.js 15 App Router, which serves as the backbone for both the user interface and the API gateway to the local AI services. This section provides a deep dive into the structural patterns for building a multi-panel dashboard, a definitive decision framework for selecting between Server Actions and Route Handlers, and practical TypeScript implementations for handling the critical streaming of LLM responses from the Ollama backend. The goal is to establish a scalable, maintainable, and performant architecture that leverages modern Next.js features while navigating potential limitations of the Edge Runtime environment.

To accommodate the Command Center's diverse functionalities—chat, RAG playground, agent management, and hardware monitoring—a sophisticated layout is required. The Next.js App Router provides two powerful primitives for this purpose: **route groups** and **parallel routes**. Route groups, denoted by `(group-name)/`, allow for shared layouts without altering the URL structure. This is ideal for creating a consistent top navigation bar, sidebar, and footer that remain constant across all views [[4]]. For instance, `(dashboard)/layout.tsx` could define the overarching structure of the Command Center. Parallel routes, created using the syntax `(...panel-name)/`, enable the simultaneous rendering of multiple independent route segments. This is the key to the Command Center's multi-pane interface. For example, the root layout could define a primary content area as `<div className="flex-1"><Outlet /></div>`, where `<Outlet />` renders the parallel route content. One parallel route could be `(...chat)/page.tsx` for the chat interface, another `(...rag)/page.tsx` for the RAG playground, and so on. This structure allows the user to switch between panels without reloading the entire page or losing state in other parts of the UI, creating a fluid, single-page application experience [[4]].

Once the layout is established, the next critical decision is determining the appropriate mechanism for API communication: Server Actions or Route Handlers. This is not a matter of preference but a strategic choice based on the nature of the request. **Server Actions** are best suited for discrete, UI-related mutations and form submissions. They allow developers to write server-side code directly within React components, simplifying the process of updating state on the server in response to user actions, such as sending a new message or saving a model configuration [[64]]. Their declarative nature and automatic integration with React's error boundaries make them ideal for these types of operations [[65]]. Conversely, **Route Handlers** offer a more granular and flexible approach for creating custom API endpoints. They are the preferred method for any operation that doesn't fit the simple mutation model, particularly those involving long-lived connections or complex data manipulation [[44]]. All streaming operations, such as fetching a continuous stream of tokens from Ollama, must be implemented using Route Handlers because they have direct access to the `Request` and `Response` objects, enabling the creation of a `ReadableStream` for streaming data [[32]].

Therefore, a hybrid architecture is optimal for the Command Center:

- Use **Server Actions** for short-lived, state-changing interactions like submitting a prompt, switching a model, or toggling a setting.
- Use **Route Handlers** for all long-running processes, especially the primary `/api/chat/stream` endpoint responsible for connecting to Ollama's streaming API.

The implementation of this streaming endpoint in a Route Handler is central to the user experience. The process involves creating a file like `app/api/chat/route.ts` and exporting an asynchronous POST function. Inside this function, the incoming request body (containing the prompt and selected model) is parsed. A connection is then established with the local Ollama server using a library like the official `ollama` package [[70]]. The `generate` or `chat` endpoint is called with the `stream: true` option. Ollama will respond with an HTTP response whose body is a `ReadableStream` of text, with each chunk containing a single token [[31]]. The Route Handler's own response object is configured to also be a `ReadableStream`. The handler then pipes the stream from Ollama directly into its response stream, effectively proxying the data back to the client in real-time. This pattern ensures minimal latency and efficient memory usage.

For developers looking to streamline this process, the Vercel AI SDK offers a powerful abstraction. It provides React components like `useChat` that handle the complexities of streaming under the hood [[5]]. Crucially, the SDK can be configured to point to a local Ollama instance by setting the `provider` to a custom provider and specifying the base URL [[71,73]]. The `ai-sdk-ollama` package is a pre-built provider that simplifies this configuration, offering a type-safe and future-proof connection to Ollama's API [[71]]. Using the AI SDK is highly recommended as it reduces boilerplate code and aligns the project with Vercel's ecosystem best practices. A typical configuration would involve initializing the AI SDK in the application root with a custom provider pointing to `http://localhost:11434` (the default Ollama port), allowing components like `useChat` to seamlessly communicate with the local model runtime [[33]].

However, developers must be mindful of the environment in which this code executes. While Next.js 15 encourages migration to the App Router and modern APIs [[4]], certain packages or functions may encounter limitations in the Edge Runtime, which is optimized for global distribution and serverless functions [[3]]. Although the Vercel AI SDK is designed to work in these environments, it is always prudent to test thoroughly. If issues arise, deploying the API routes to the Node.js-based Server Runtime (the default for Vercel deployments) is a viable fallback, ensuring broader compatibility with Node.js-native libraries like `systeminformation` [[45]]. The final architecture must therefore consider the trade-offs between the performance benefits of the Edge Runtime and the broader compatibility of the Server Runtime.

## Multi-Model Orchestration and RAG Pipeline Design

The intellectual core of the Command Center lies in its ability to intelligently orchestrate multiple local AI models and leverage Retrieval-Augmented Generation (RAG) to provide contextually grounded responses. These features elevate the application from a simple chat wrapper to a powerful developer-grade tool. This section details the architectural patterns for balancing model selection based on latency, capability, and task type; managing context across model switches; and constructing a fully local RAG pipeline from document ingestion to hybrid search. Every decision is analyzed for its trade-offs, and concrete implementation patterns are provided.

**Multi-Model Orchestration** requires moving beyond static model selection. A sophisticated router must dynamically choose the best model for a given task. The user's requirement for a balance between latency, capability, and task type suggests a multi-faceted routing logic. Three distinct designs can be considered:

1.  **Capability-Based Routing:** This logic inspects the task and selects a model that possesses the required capabilities. For instance, if a prompt contains a request for code generation or uses a specific JSON format, the router would consult a model registry to find models known to support function calling or structured output. This registry can be populated by querying Ollama's `/api/tags` endpoint, which provides metadata about each loaded model [[34]]. This approach ensures that tasks requiring specific abilities (like math or tool use) are routed to capable models, improving reliability.

2.  **Latency-Based Routing:** This design prioritizes speed. It maintains a record of the average response time for each model and directs simpler, less complex prompts to the fastest available model. This is particularly useful for tasks like summarization or simple Q&A where cutting-edge reasoning is not required. The trade-off is that it may sacrifice quality for speed, and maintaining accurate latency metrics adds a layer of operational complexity.

3.  **Task-Type-Aware Hybrid Routing:** This represents the most advanced pattern, combining the first two. The router first categorizes the prompt (e.g., using a lightweight local classifier or keyword matching). Based on the category, it applies a set of rules. For a "coding" task, it would filter for models with strong coding capabilities (from the capability registry) and then select the one with the lowest latency among that subset. This balanced approach provides the highest degree of optimization, ensuring both correctness and performance.

A critical challenge in multi-model orchestration is **context window management**. When a conversation's context exceeds the capacity of the current model, a switch is necessary. Simply truncating the context is lossy. A better strategy is to summarize the older context before switching, or to pass a limited amount of the most recent conversation history along with a summary of the rest. This requires careful state management in the application to track the total context size and implement a summarization step, perhaps using a small, fast model like Phi-3 or TinyLlama, before invoking the larger target model. Libraries like `unified-llm/core` demonstrate how to create a unified interface for multiple providers, which can serve as a conceptual model for managing these transitions [[74]].

The second pillar of the Command Center's intelligence is the **RAG Pipeline**. A complete local RAG architecture consists of several stages:

- **Document Ingestion:** The pipeline must accept various document formats (PDF, DOCX, TXT, HTML). A robust solution involves using a parsing library like `pdf-parse` or `mammoth` to extract raw text. The quality of the RAG system depends heavily on the chunking strategy. For code documents, smaller chunks preserving syntactic blocks are preferable. For technical documentation or prose, larger chunks based on semantic sentences or paragraphs are better. A sliding window approach can be used to create overlapping chunks, helping to preserve context across boundaries [[61]].
- **Embedding Generation:** The extracted text chunks must be converted into numerical vectors. This requires a local embedding model served via Ollama. Models like `nomic-embed-text`, `mxbai-embed-large`, or Snowflake's `arctic-embed` are strong candidates that can run on consumer hardware [[77]]. The choice of model involves a trade-off between performance, accuracy, and resource consumption.
- **Vector Storage:** The generated vectors need to be stored in a vector database. Several options exist:
  - **ChromaDB:** An easy-to-use, open-source vector database with a simple Python API. It can be run as a local server and accessed via a REST API, making it compatible with the Node.js backend.
  - **LanceDB:** Another file-based vector database that is lightweight and performs well for local applications.
  - **PGvector:** An extension for PostgreSQL that adds native support for vector data types and similarity search. Since the Command Center's persistence layer will use a relational database, using PGvector via PGlite is a compelling option as it keeps the stack unified [[58,68]].
  - **Qdrant:** A high-performance vector search engine written in Rust, known for its speed and rich feature set.

  A comparison table helps clarify the choices.

| Vector Store | Pros                                                                         | Cons                                                                            | Best Fit for Command Center                                                     |
| :----------- | :--------------------------------------------------------------------------- | :------------------------------------------------------------------------------ | :------------------------------------------------------------------------------ |
| **ChromaDB** | Simple to set up, good Python/JS clients, actively maintained.               | Separate service dependency, Python-centric ecosystem.                          | Good for rapid prototyping or if a pure-JS stack is desired.                    |
| **LanceDB**  | File-based, very lightweight, high performance for local use.                | Less mature than Chroma, smaller community.                                     | Excellent choice for a standalone, self-contained application.                  |
| **PGvector** | Unified stack (uses same DB as app state), powerful SQL integration, mature. | Higher resource overhead than file-based stores, requires PostgreSQL knowledge. | Recommended if leveraging PGlite for the main persistence layer.                |
| **Qdrant**   | Extremely fast, rich feature set (filtering, payload storage), Rust-based.   | More complex to deploy and manage locally.                                      | Overkill for MVP, but a strong option if raw search speed becomes a bottleneck. |

- **Hybrid Search:** To achieve the best retrieval results, a hybrid approach combining keyword-based (BM25) and semantic (vector) search is essential. The logic for this resides in the Next.js API layer. When a query is received, it is first processed through a local BM25 search algorithm against the document text. Simultaneously, the query is embedded and used to perform a vector similarity search on the chosen vector store. The results from both searches are then combined using a weighted score, giving more importance to relevance (vector search) while ensuring keyword matches are not missed. This fusion of techniques significantly improves the recall and precision of the RAG system [[47]].
- **Re-ranking:** After the initial retrieval, a re-ranking step can further improve answer quality. Instead of just retrieving the top-k documents, the retrieved set is passed back to an LLM (which can be a smaller, faster model) along with the original query. The LLM is prompted to re-rank the documents based on their relevance. This "re-ranker" model acts as a final filter, ensuring the most pertinent information is fed into the final generation step. This entire process—from ingestion to re-ranking—must be executed locally to comply with the project's core constraint of no cloud dependencies.

## Agent Systems and Real-Time Monitoring

The Command Center's ambition extends to hosting autonomous agents and providing deep visibility into the system's performance through real-time hardware monitoring. These features represent the pinnacle of a developer-grade local AI control surface. This section outlines the architecture for integrating Model Context Protocol (MCP) based agents, detailing the setup of a local MCP server, the implementation of a ReAct loop in TypeScript, and the design of a robust task queue. It also provides a complete blueprint for the real-time monitoring panel, covering metric collection, transport protocols, and visualization.

**Integrating MCP-based Agent Systems** transforms the Command Center from a reactive interface into a proactive workflow manager. The Model Context Protocol (MCP) is an emerging standard for enabling LLMs to interact with external tools and services, similar to function calling but with a more formalized structure [[52]]. The architecture involves several key components:

- **Local MCP Server:** An MCP server must be set up to run alongside the Next.js application. Boilerplates and templates, such as the one from LobeHub or the TypeScript template from LobeChat, provide a solid starting point for building a remote MCP server [[7,8]]. This server will expose a set of tools that the LLM agents can discover and use.
- **Tool Registry:** A central registry must be implemented to manage the tools available to agents. This can be a simple TypeScript object or a more structured database table. Each tool definition should include a name, a description for the LLM, and the actual function to execute. The MCP server communicates this registry to connected LLMs, allowing them to understand what actions they can take [[30]].
- **ReAct Loop Implementation:** The core of the agent's intelligence is the ReAct (Reason+Act) loop, implemented within the Next.js API layer. The TypeScript logic would follow a cycle: 1) Receive a user goal. 2) Send the goal and available tools to an LLM (e.g., Llama 3, Mistral) and prompt it to generate a plan and the next action (e.g., "Use 'search_wiki' with query='quantum computing'"). 3) Parse the LLM's output, execute the corresponding function, and capture the result. 4) Append the observation (tool output) to the conversation history and repeat the process until the goal is achieved. This iterative reasoning loop is fundamental to agent-based problem solving [[30]].
- **Agent Task Queue:** Since agent tasks can be long-running and asynchronous, they must be managed in a persistent queue. SQLite is an excellent choice for this due to its file-based nature, zero-configuration setup, and suitability for a local application. A dedicated table, `agent_tasks`, would be needed to track the state of each job. The schema for this table is critical for managing retries, failures, and progress tracking.

The following SQLite schema defines the `agent_tasks` table:

| Column Name            | Data Type | Constraints                 | Description                                                  |
| :--------------------- | :-------- | :-------------------------- | :----------------------------------------------------------- |
| `id`                   | INTEGER   | PRIMARY KEY, AUTOINCREMENT  | Unique identifier for the task.                              |
| `task_name`            | TEXT      | NOT NULL                    | Descriptive name for the task (e.g., 'Web Research').        |
| `status`               | TEXT      | NOT NULL, DEFAULT 'pending' | Current state ('pending', 'running', 'succeeded', 'failed'). |
| `input_params`         | JSON      | NOT NULL                    | JSON object containing the initial parameters for the task.  |
| `conversation_history` | JSON      | NULL                        | Stores the evolving dialogue between the agent and tools.    |
| `current_step`         | INTEGER   | DEFAULT 0                   | Index of the current step in the ReAct loop.                 |
| `max_steps`            | INTEGER   | DEFAULT 10                  | Maximum number of steps allowed for the agent.               |
| `retry_count`          | INTEGER   | DEFAULT 0                   | Number of times the task has been retried after failure.     |
| `max_retries`          | INTEGER   | DEFAULT 3                   | Maximum number of retry attempts.                            |
| `created_at`           | DATETIME  | DEFAULT CURRENT_TIMESTAMP   | Timestamp of when the task was created.                      |
| `updated_at`           | DATETIME  | DEFAULT CURRENT_TIMESTAMP   | Timestamp of the last update to the task's status.           |
| `result`               | TEXT      | NULL                        | Final output or result of the successful task.               |
| `error_message`        | TEXT      | NULL                        | Error message if the task failed after all retries.          |

This schema provides a robust foundation for managing asynchronous agent execution, including idempotency, retry logic, and state persistence [[85]].

Parallel to the complexity of agent systems, the **Real-Time Monitoring Panel** provides essential operational insight. This panel must visualize the consumption of critical hardware resources. The architecture for this panel is straightforward:

- **Metric Collection Strategy:** The `systeminformation` npm package is the cornerstone of this system. It provides a cross-platform way to gather detailed hardware metrics, including CPU load, RAM usage, and, crucially, VRAM usage for GPUs [[83]]. A scheduled job (e.g., using `setInterval`) within the Next.js API layer should poll these metrics at regular intervals (e.g., every 1-2 seconds). The data collected should include CPU percentage, total and free RAM, and total and free VRAM. While Ollama's `/api/ps` endpoint can provide information about running containers, it does not offer the granular, real-time telemetry on resource utilization needed for a responsive dashboard; therefore, relying on `systeminformation` is the recommended approach [[34]].
- **Transport Protocol Recommendation:** For pushing these telemetry updates from the server to the frontend, **Server-Sent Events (SSE)** is the superior choice over WebSockets. SSE operates over a single, standard HTTP connection and is unidirectional (server-to-client), perfectly matching the use case of a dashboard that only needs to receive updates [[31,39]]. It is simpler to implement, handles connection drops gracefully, and is natively supported by browsers. WebSockets, while bidirectional, introduce unnecessary complexity and a persistent TCP connection for a simple data streaming scenario [[29]].
- **Charting Library Recommendation:** For visualizing this streaming data, Tremor is a strong candidate. It is a modern, performant, and extensible toolkit for creating interactive data visualizations in the browser, designed to handle streaming data efficiently [[3]]. Its declarative API and focus on real-time dashboards make it a suitable choice. Alternatively, libraries like Observable Plot or even Recharts, configured for streaming updates, could also be effective. The key requirement is that the library can efficiently append new data points to a chart without causing performance degradation during high-frequency updates.

By combining a sophisticated agent framework with a transparent monitoring system, the Command Center gains the power and observability required for serious development and experimentation with local AI.

## Persistence, Security, and Operational Tooling

Beyond the core AI functionalities, a production-ready Command Center requires a robust persistence layer, a hardened security posture, and comprehensive tooling for managing prompts and evaluating performance. This section addresses the critical decisions around the local database, outlines a strategy for securing an air-gapped deployment, and provides detailed blueprints for prompt versioning and evaluation systems. These elements ensure the application is durable, secure, and conducive to iterative improvement.

The choice of a local persistence layer is a fundamental architectural decision. The project scope specifies three primary options: Drizzle ORM with SQLite, Prisma with SQLite, or PGlite (a headless Postgres instance) [[58]]. Each presents a unique set of trade-offs.

- **Drizzle ORM with SQLite:** Drizzle is celebrated for its compile-time type safety, generating TypeScript types directly from the database schema. This eliminates runtime type errors related to database queries and provides an exceptional developer experience for writing safe, auto-completed queries. It is also extremely lightweight, resulting in fast cold starts and minimal bundle sizes, making it ideal for edge environments or standalone applications [[1,3]]. Its flexibility allows it to be used for both schema management and runtime queries, fitting well within a Next.js App Router project [[3]].
- **Prisma with SQLite:** Prisma offers a highly opinionated and ergonomic ORM with a powerful data modeling language (Prisma Schema) and a rich set of features out of the box, such as migrations and seeding. It is often praised for its simplicity and "ease of use," making it a popular choice for enterprise systems [[2]]. While it may have slightly higher overhead than Drizzle, its maturity and extensive documentation make it a very safe and stable choice.
- **PGlite:** PGlite provides a near-complete, headless PostgreSQL experience. Its primary advantage is the full power of SQL, including advanced features like the PGvector extension for vector similarity search, foreign data wrappers, and complex transactions [[58,68]]. This makes it the most powerful option for complex relational data and advanced querying. However, this power comes with increased complexity and resource consumption compared to a file-based SQLite database.

Given the Command Center's requirements—a standalone application needing to store diverse data types (conversations, agent logs, prompt evaluations) and perform similarity searches for RAG—the final recommendation is **Drizzle ORM with SQLite**. It offers the best balance of compile-time type safety, performance, and developer experience for this use case [[1]]. The simplicity of SQLite avoids the overhead of a full PostgreSQL instance, while Drizzle's type safety mitigates the risk of common database-related bugs. If the need for advanced PostgreSQL features or a massive scale-out scenario ever arises, migrating from Drizzle/SQLite to Prisma/PostgreSQL or directly to PGlite would be a feasible future evolution.

The following entity schema, defined using Drizzle's schema builder syntax, covers all six required data types:

```typescript
// entities.ts
import { integer, text, sqliteTable, primaryKey, json } from 'drizzle-orm/sqlite-core'

// Conversations table
export const conversations = sqliteTable('conversations', {
  id: integer('id').primaryKey({ autoIncrement: true }),
  title: text('title').notNull(),
  createdAt: text('created_at').default(() => 'CURRENT_TIMESTAMP'),
})

// Messages table
export const messages = sqliteTable('messages', {
  id: integer('id').primaryKey({ autoIncrement: true }),
  conversationId: integer('conversation_id')
    .notNull()
    .references(() => conversations.id),
  sender: text('sender', { enum: ['user', 'assistant'] }).notNull(), // 'user' or 'assistant'
  textContent: text('text_content'),
  toolCalls: json('tool_calls'), // Array of tool call objects
  createdAt: text('created_at').default(() => 'CURRENT_TIMESTAMP'),
})

// Model Configurations table
export const modelConfigs = sqliteTable('model_configs', {
  id: integer('id').primaryKey({ autoIncrement: true }),
  modelName: text('model_name').notNull().unique(),
  baseUrl: text('base_url').default('http://localhost:11434'),
  settings: json('settings'), // JSON object for model-specific params
})
```

**Security and Network Isolation** are paramount for an application handling potentially sensitive data, especially in an air-gapped environment. The first line of defense is to strictly limit the attack surface of the Ollama runtime. Ollama must be configured to bind its API exclusively to `localhost` (127.0.0.1). This is achieved by setting the `OLLAMA_HOST` environment variable to this address before starting the service [[33]]. This prevents any external network access to the model server, ensuring that models and data remain contained within the local machine. For added security in enterprise contexts, firewalls and network segmentation can be employed [[88]].

While Ollama itself is the primary service, the Next.js frontend may need to connect to other local services (e.g., a separate vector database server). All such connections must be explicitly managed and secured. Furthermore, for collaborative or multi-user scenarios, a local authentication system is necessary. NextAuth.js v5 provides a robust solution for this, specifically its credentials provider, which allows for managing user logins with email and password stored locally, without relying on external identity providers like Google or GitHub [[90]]. This creates a closed-loop authentication system that complements the air-gapped nature of the application. Sensitive environment variables, such as database passwords or API keys for any external-but-local services, must be managed securely using `.env` files or a secrets management tool within the deployment environment.

Finally, the Command Center must include tooling for **Prompt Pipeline Management**. This is crucial for reproducibility, A/B testing, and continuous improvement. A versioned prompt template system can be implemented using a TypeScript interface that defines the structure of a prompt, including placeholders for variables.

```typescript
// PromptTemplate.ts
interface PromptVersion {
  version: string // e.g., "v1.2.1"
  changelog: string
  createdAt: Date
  schema: {
    [key: string]: { type: string; description: string } // e.g., { "document": { type: "string", description: "The text to summarize" } }
  }
  template: string // e.g., "Summarize the following document. Document: {{document}}"
}

type PromptPipeline = {
  id: string
  name: string
  latestVersion: string
  versions: PromptVersion[]
}
```

Local evaluation scoring can be performed using frameworks like RAGAS, which can be run in a separate Node.js process to assess generated answers against ground-truth data [[25]]. A/B testing can be tracked by assigning a unique experiment ID to each prompt version and logging which version was used for each run in the `promptEvaluations` table. This data forms the basis for a feedback loop, allowing developers to compare the performance of different prompt versions and refine them over time.

## Phased Development Roadmap and Risk Mitigation

Developing a comprehensive local AI Command Center is a complex undertaking that benefits from a structured, incremental approach. A phased roadmap allows for the delivery of value at each stage, validates core assumptions early, and mitigates the risk associated with the project's inherent technical challenges. This section outlines a four-phase build plan, detailing the deliverables, estimated complexity, key dependencies, and primary risks for each phase. It concludes with an analysis of the highest-risk architectural decision and two credible mitigation paths.

**Phase 1: Foundational Elements and Core Chat Interface (Complexity: M)**

The initial phase focuses on establishing the core functionality: a working chat interface that can communicate with the local Ollama runtime. This serves as the Minimum Viable Product (MVP) and validates the most critical integration path.

- **Deliverables:**
  - A functional Next.js 15 application with a clean UI for chat.
  - A parallel route architecture for the chat panel within a dashboard layout [[4]].
  - A Route Handler configured to proxy requests to the local Ollama API (`http://localhost:11434`) [[44]].
  - A streaming implementation that fetches and displays LLM responses token by token using Server-Sent Events (SSE) [[31]].
  - Basic model switching functionality, allowing the user to select a model from the list returned by Ollama's `/api/tags` endpoint [[34]].
  - A basic SQLite database schema for storing conversation history [[1]].
- **Key Dependencies:** Next.js 15, `ollama` NPM package, Drizzle ORM, SQLite.
- **Primary Risk:** The complexity of setting up and debugging the streaming connection between the Next.js Route Handler and the Ollama server. Potential issues could arise from differences in how streams are handled in the Next.js environment (Edge vs. Server Runtime) or from Ollama's specific streaming response format [[32]].

**Phase 2: RAG Pipeline Integration (Complexity: L)**

With the core chat interface established, Phase 2 expands the application's capabilities by integrating a full Retrieval-Augmented Generation pipeline. This significantly enhances the utility of the Command Center for knowledge-based tasks.

- **Deliverables:**
  - A document ingestion UI for uploading files (PDF, TXT, etc.) [[51]].
  - A backend pipeline for parsing documents and applying a chunking strategy [[61]].
  - Integration with a local embedding model (e.g., `nomic-embed-text`) via Ollama [[77]].
  - A choice of vector store (e.g., ChromaDB or LanceDB) configured and accessible from the Next.js backend [[58]].
  - A "Playground" panel with a query box that performs hybrid BM25 + vector search and returns relevant document chunks [[47]].
  - An updated conversation interface that can use retrieved context to inform the LLM's responses.
- **Key Dependencies:** `pdf-parse` or similar parser, a local embedding model, a vector database library (e.g., `chromadb`), a BM25 search implementation.
- **Primary Risk:** Performance bottlenecks in the RAG pipeline. Locally running embedding models and vector searches can be computationally expensive, leading to high latency for users. The choice of chunking strategy and vector index size will have a significant impact on both performance and quality [[61]].

**Phase 3: Agent and MCP Framework (Complexity: XL)**

This phase introduces the most advanced feature: autonomous agents powered by the Model Context Protocol. This represents a major leap in complexity and transforms the Command Center into a true workflow engine.

- **Deliverables:**
  - A local MCP server set up and running alongside the Next.js application, using a boilerplate like the one from LobeHub [[8]].
  - A TypeScript-based tool registry for defining and exposing functions to agents [[7]].
  - An implementation of a ReAct loop in TypeScript within the API layer [[30]].
  - A persistent agent task queue using SQLite, with the schema defined in the previous section [[85]].
  - A dedicated "Agents" panel for creating, monitoring, and viewing the logs of running agent tasks.
- **Key Dependencies:** MCP TypeScript SDK, a job queue library for SQLite, a set of functional tools (e.g., web searcher, calculator).
- **Primary Risk:** Complexity of the orchestration logic. The interaction between the user, the ReAct loop, the tool registry, and the persistent queue creates a complex state machine that is difficult to debug and maintain. Ensuring reliable error handling and retry logic for long-running agent tasks is a significant challenge [[85]].

**Phase 4: Monitoring and Evaluation Suite (Complexity: M)**

The final phase focuses on enhancing the developer experience with tools for observing system performance and measuring the effectiveness of the AI components.

- **Deliverables:**
  - A real-time hardware monitoring dashboard panel.
  - Integration of the `systeminformation` npm package to collect CPU, RAM, and VRAM metrics [[83]].
  - An SSE endpoint to stream these metrics to the frontend dashboard.
  - A "Prompt Management" panel for versioning prompt templates and conducting A/B tests.
  - A local evaluation scoring system, potentially using a framework like RAGAS [[25]].
  - A comprehensive `promptEvaluations` table in the database to log and analyze test results [[1]].
- **Key Dependencies:** `systeminformation` NPM package, a charting library for streaming data (e.g., Tremor), a local evaluation framework.
- **Primary Risk:** Inaccurate or unreliable hardware metrics. The ability of `systeminformation` to correctly report VRAM usage can be inconsistent across different GPU vendors (NVIDIA, AMD) and operating systems (macOS, Linux), which could undermine the value of the monitoring panel [[83]].

### Architect's Verdict

The highest-risk architectural decision is the choice of **Drizzle ORM with SQLite as the primary persistence layer**. While it offers an excellent balance of type safety, performance, and developer experience, its limitations in handling complex queries, particularly for advanced vector similarity searches, could become a significant bottleneck as the application scales in complexity and data volume. The current schema is sufficient for basic needs, but as the RAG and evaluation systems become more sophisticated, the lack of native, high-performance vector search capabilities (as found in PGvector) could lead to unacceptable performance degradation.

Two credible mitigation paths exist:

1.  **Stick with the Drizzle/SQLite Stack:** This path involves accepting the performance limitations and optimizing the application logic accordingly. This could include caching query results, using approximate nearest neighbor (ANN) algorithms where possible, and carefully managing the size of the datasets being queried. This is the lower-risk path in terms of implementation effort but carries the risk of hitting a hard performance wall.
2.  **Adopt PGlite for Advanced Queries:** This path involves augmenting the existing stack. The application would continue to use Drizzle/SQLite for all relational data (conversations, user settings, agent logs) but would use a separate PGlite instance solely for the vector search component. This would provide the full power of PGvector for similarity queries while retaining the simplicity of SQLite for everything else. This is a more complex architecture but offers a much higher ceiling for performance and scalability.

The **recommended path is the second one: adopting PGlite for advanced queries.** The justification is rooted in the project's long-term goals. The Command Center is intended to be a "developer-grade" tool, implying a need for robust, scalable, and high-performance features from day one. By investing in PGlite early on, the architecture is future-proofed against the limitations of SQLite's query performance. The added complexity of managing a second database instance is a manageable trade-off for the significant gain in analytical capability and performance, ensuring the application can handle increasingly demanding RAG and evaluation workloads without compromise.
