## 1. Architecture Overview

The command center is divided into four decoupled layers. Each layer can be upgraded or replaced independently.
```

┌───────────────────────────────────────────────────────┐
│ UI LAYER — Next.js 15 (static export) + React + TS │
│ Tailwind · Shadcn/ui · Vercel AI SDK · Zustand │
├───────────────────────────────────────────────────────┤
│ SHELL LAYER — Tauri 2.0 (Rust) │
│ Window mgmt · System tray · File system · OS IPC │
├───────────────────────────────────────────────────────┤
│ RUNTIME LAYER — Node.js Sidecar │
│ LanceDB · Embeddings · MCP tool server · Ingestion │
├───────────────────────────────────────────────────────┤
│ INFERENCE LAYER — Ollama (localhost:11434) │
│ REST API · Model mgmt · Streaming · Tool calls │
└───────────────────────────────────────────────────────┘

````

**Verified facts:**
- Tauri does not include a Node.js runtime in the webview; all server-side logic must live in the Rust core or a sidecar.
- The UI communicates with Ollama directly via `fetch()` to `localhost:11434` — no Rust bridge needed for inference.
- The Node sidecar handles everything that requires Node-native modules: LanceDB, embeddings, MCP bridges, file ingestion.
- Rust (Tauri core) handles only what needs OS-level access: file-system, notifications, system tray, window lifecycle.

|---|---|---|---|
| Desktop shell | **Tauri 2.0** | Electron | ~10MB bundles vs Electron's 100MB+; ~50MB RAM vs ~200MB |

| Styling | **Tailwind CSS + Shadcn/ui** | MUI, Chakra | Matches existing agency stack |

| Monorepo | **Turborepo** | Nx | Familiar in Next.js ecosystem; fast caching |
| MCP bridge | **`tauri-plugin-mcp-bridge`** | Custom IPC | Exposes Tauri IPC and backend state over MCP |

## 3. Tauri + Next.js Setup

### Verified behavior

Next.js **must** be configured as a static export for Tauri because Tauri's webview has no Node.js runtime. This disables:
- Server Actions
- API Routes
- Incremental Static Regeneration (ISR)
- Middleware

Everything that would normally live in a Next.js server must move to either the Rust core (`invoke`) or the Node sidecar (HTTP/IPC).

### `next.config.ts`

```ts
/** @type {import('next').NextConfig} */
const nextConfig = {
  output: 'export',          // forces SSG/SPA; required for Tauri
  images: { unoptimized: true }, // required for static export
  trailingSlash: true,       // optional but useful for client-side routing
}
export default nextConfig
````

### `tauri.conf.json` (relevant section)

```json
{
  "build": {
    "frontendDist": "../out",
    "devUrl": "http://localhost:3000"
  }
}
```

### Dev vs. production mode

| Mode        | Next.js command              | Tauri command                        |
| ----------- | ---------------------------- | ------------------------------------ |
| Development | `next dev`                   | `tauri dev` (reads `devUrl`)         |
| Production  | `next build` → writes `out/` | `tauri build` (reads `frontendDist`) |

### Dynamic routing

You can still use client-side dynamic routes (e.g., `/chat/[id]`), but you must:

```ts
// app/chat/[id]/page.tsx
export function generateStaticParams() {
  return [] // must exist; empty is fine for client-only dynamic routes
}
```

## 4. Inference Layer — Ollama

### Verified REST API endpoints

| Endpoint          | Method | Purpose                                         |
| ----------------- | ------ | ----------------------------------------------- |
| `/api/tags`       | GET    | List all locally pulled models                  |
| `/api/show`       | POST   | Inspect model metadata, architecture, Modelfile |
| `/api/pull`       | POST   | Pull a model with streamed progress             |
| `/api/delete`     | DELETE | Remove a model                                  |
| `/api/chat`       | POST   | Chat inference (streaming or not)               |
| `/api/generate`   | POST   | Raw completion inference                        |
| `/api/embeddings` | POST   | Embed text via a model                          |

### Streaming behavior (verified)

- Streaming is **enabled by default over the REST API** but **disabled by default in SDKs**.
- You must pass `stream: true` explicitly in SDK calls.
- Each streamed chunk is a newline-delimited JSON object.
- When `done: true` is received, the chunk also contains `eval_count` (tokens generated) and `total_duration` (nanoseconds) — useful for your stats panel.
- Thinking mode emits a separate `thinking` field in each chunk; `content` accumulates the final assistant response.

### Official `ollama-js` streaming pattern

```ts
import ollama from 'ollama'

const stream = await ollama.chat({
  model: 'llama3.2',
  messages: [{ role: 'user', content: 'Hello' }],
  stream: true,
})

for await (const chunk of stream) {
  process.stdout.write(chunk.message.content)
}
```

### Tool calling (streaming)

Tool calls are emitted as `tool_calls` fields in streamed chunks. Ollama added streaming responses with tool calling in 2025. Design your tool-calling layer to:

1. Accumulate `tool_calls` across chunks
2. Execute the tool when `done: true`
3. Append the tool result as a `tool` role message
4. Re-invoke the model for a final response

### Model Persona (Modelfile)

Store reusable personas in SQLite as raw Modelfile strings, then create named models at runtime:

```
FROM llama3.2:3b
SYSTEM "You are an expert Next.js developer and code reviewer."
PARAMETER temperature 0.4
PARAMETER num_ctx 8192
PARAMETER top_k 40
```

Call `POST /api/create` to instantiate a persona as a named model, or inject the system prompt at the chat API level for ephemeral personas.

## 5. Vercel AI SDK Integration

### Verified providers

| Package                 | Notes                                                                                   |
| ----------------------- | --------------------------------------------------------------------------------------- |
| `ollama-ai-provider-v2` | Implements Ollama REST; supports streaming, embeddings, tool calling, `providerOptions` |
| `ai-sdk-ollama`         | Vercel AI SDK v5+ provider; built on `ollama` JS library; type-safe                     |

Both are valid. `ollama-ai-provider-v2` gives more fine-grained `providerOptions` control.

### Setup

```ts
import { streamText, tool } from 'ai'
import { createOllama } from 'ollama-ai-provider-v2'
import { z } from 'zod'

const ollama = createOllama({ baseURL: 'http://localhost:11434/api' })

const { textStream } = streamText({
  model: ollama('llama3.2', {
    providerOptions: {
      ollama: {
        options: {
          num_ctx: 8192,
          top_k: 40,
          temperature: 0.7,
        },
      },
    },
  }),
  messages,
  tools: {
    readFile: tool({
      description: 'Read a file from the local filesystem',
      inputSchema: z.object({ path: z.string() }),
      execute: async ({ path }) => {
        // Call Tauri's Rust backend via invoke()
        return await invoke('read_file', { path })
      },
    }),
  },
})
```

### Shared hooks across web and desktop

Because the Vercel AI SDK is web-compatible, you can define a `packages/llm-hooks` package in your monorepo that exports:

```ts
// packages/llm-hooks/useLocalChat.ts
export function useLocalChat(model: string, systemPrompt?: string) {
  return useChat({
    api: undefined, // omit for direct Ollama calls
    experimental_prepareRequestBody: ({ messages }) =>
      buildOllamaPayload(model, messages, systemPrompt),
  })
}
```

Both your desktop app and web dashboard consume the same hook.

## 6. Node.js Sidecar Strategy

### Why a sidecar is needed

The Tauri webview cannot import Node-native modules (`@lancedb/lancedb`, `fs`, `crypto`, etc.) because the frontend runtime is a browser, not Node.js. A Node sidecar solves this by:

- Running a lightweight Fastify/Express server on a local port
- Exposing REST endpoints for embedding, ingestion, RAG queries, and tools
- Being bundled into the Tauri binary as an `externalBin`

### Sidecar architecture

```
Tauri App
 ├── Rust core (Tauri)
 │   └── spawns sidecar on startup via shell plugin
 │
 └── Node sidecar (Fastify server)
     ├── POST /embed         → calls ollama.embeddings()
     ├── POST /ingest        → chunks, embeds, writes to LanceDB
     ├── POST /search        → queries LanceDB, returns top-k chunks
     ├── GET  /tools         → returns MCP-style tool schemas
     └── POST /tools/:name   → executes a named tool
```

### Bundling

Use `pkg` or `esbuild + ncc` to compile your Node server into a standalone binary:

```bash
npx pkg sidecar/index.js --target node18-macos-arm64 --output sidecar-aarch64-apple-darwin
```

Register it in `tauri.conf.json`:

```json
{
  "bundle": {
    "externalBin": ["binaries/sidecar"]
  }
}
```

Tauri expects platform-specific filenames using the `-<target-triple>` convention (e.g., `sidecar-aarch64-apple-darwin`, `sidecar-x86_64-pc-windows-msvc`).

### Spawning from Rust

```rust
use tauri_plugin_shell::ShellExt;

fn start_sidecar(app: &tauri::AppHandle) {
  let _ = app.shell()
    .sidecar("sidecar")
    .unwrap()
    .spawn();
}
```

### Key practical notes

- Bundle only what you need; use `esbuild` to tree-shake and eliminate unused native modules.
- The additional disk footprint is ~30–60MB for a Node binary + your server code — acceptable for a desktop app.
- Use `tauri-plugin-shellx` only in development; in production, use the strict `shell` plugin with a scoped allow-list.

## 7. Local Persistence — SQLite

### Schema design

Use `tauri-plugin-sql` (wraps `rusqlite`) for all persistent state.

```sql
-- sessions
CREATE TABLE sessions (
  id TEXT PRIMARY KEY,
  title TEXT,
  model TEXT NOT NULL,
  persona_id TEXT,
  created_at INTEGER NOT NULL,
  updated_at INTEGER NOT NULL
);

-- messages
CREATE TABLE messages (
  id TEXT PRIMARY KEY,
  session_id TEXT NOT NULL REFERENCES sessions(id),
  role TEXT NOT NULL,         -- 'user' | 'assistant' | 'tool'
  content TEXT NOT NULL,
  tool_calls TEXT,            -- JSON, nullable
  thinking TEXT,              -- nullable
  created_at INTEGER NOT NULL
);

-- rolling summaries (for context window management)
CREATE TABLE summaries (
  session_id TEXT PRIMARY KEY REFERENCES sessions(id),
  content TEXT NOT NULL,
  token_count INTEGER,
  updated_at INTEGER NOT NULL
);

-- personas
CREATE TABLE personas (
  id TEXT PRIMARY KEY,
  name TEXT NOT NULL,
  modelfile TEXT NOT NULL,    -- raw Modelfile string
  base_model TEXT NOT NULL,
  created_at INTEGER NOT NULL
);

-- settings
CREATE TABLE settings (
  key TEXT PRIMARY KEY,
  value TEXT NOT NULL
);
```

### MCP context pattern

Rather than building tool results and session history into every raw inference call, expose them through MCP-style "context tools." The model calls `get_session_context(session_id)` as a tool, and the tool returns only the relevant slice from SQLite — avoiding context bloat from stuffing raw data into every message.

## 8. RAG Pipeline — LanceDB

### Why LanceDB over ChromaDB/Qdrant

| Feature                        | LanceDB | ChromaDB | Qdrant               |
| ------------------------------ | ------- | -------- | -------------------- |
| Embedded (no separate service) | ✅      | ✅       | ❌ (separate server) |
| TypeScript/Node native         | ✅      | ✅       | ✅                   |
| Rust native                    | ✅      | ❌       | ✅                   |
| File-based (portable)          | ✅      | ✅       | ❌                   |
| Apache Arrow columnar format   | ✅      | ❌       | ❌                   |
| Sub-millisecond local queries  | ✅      | ✅       | ✅                   |

LanceDB's embedded model is the strongest fit for a Tauri desktop app because it requires no installation, no Docker, and no OS-level service.

### Ingestion pipeline (Node sidecar)

```ts
import * as lancedb from '@lancedb/lancedb'
import ollama from 'ollama'

const db = await lancedb.connect('./data/lancedb')

async function ingestDocument(text: string, metadata: object) {
  const chunks = chunkText(text, { size: 512, overlap: 64 })

  const rows = await Promise.all(
    chunks.map(async (chunk, i) => {
      const { embedding } = await ollama.embeddings({
        model: 'nomic-embed-text',
        prompt: chunk,
      })
      return { text: chunk, vector: embedding, ...metadata, chunk_index: i }
    })
  )

  const table = await db.openTable('documents').catch(() => db.createTable('documents', rows))
  await table.add(rows)
}
```

### Query pipeline

```ts
async function queryRAG(question: string, topK = 5) {
  const { embedding } = await ollama.embeddings({
    model: 'nomic-embed-text',
    prompt: question,
  })

  const table = await db.openTable('documents')
  const results = await table.search(embedding).limit(topK).toArray()

  return results.map((r) => r.text).join('\n\n')
}
```

### Open WebUI RAG patterns (reference architecture)

Open WebUI uses **hybrid search** — BM25 keyword matching combined with a `CrossEncoder` re-ranker and configurable relevance thresholds. Their most impactful optimization is **chunk merging before retrieval**: reducing 588 individual chunks to ~45 semantically merged chunks cuts retrieval noise and gives the model cleaner context. You can implement a simpler version by merging adjacent chunks when their scores exceed a threshold.

## 9. MCP Integration

### What MCP gives you

Model Context Protocol (MCP) standardizes how tools expose themselves to an LLM. Instead of hardcoding tool schemas, you register MCP servers and the model discovers them dynamically. For a command center, this means:

- Each "capability" (file system, LanceDB, code runner, web search) becomes a standalone MCP server
- The model can call any registered tool by name
- Tools can return rich structured data, not just text

### Integration path: Tauri + MCP

Two Rust-side options exist:

| Package                   | Function                                                                                                         |
| ------------------------- | ---------------------------------------------------------------------------------------------------------------- |
| `mcp-tauri`               | Implements an MCP server for Tauri, exposes Tauri-specific tools (window control, file system) to MCP clients    |
| `tauri-plugin-mcp-bridge` | Bridges Tauri's IPC and backend state to MCP clients; allows external MCP clients to inspect and control the app |

### Recommended architecture

Run the MCP server inside your Node sidecar (using the official `@modelcontextprotocol/sdk`). The sidecar exposes a `stdio`-based MCP server that wraps:

- LanceDB (document search, ingestion)
- File system tools (read, write, list)
- Ollama tool proxying (e.g., image analysis, code execution)

Wire tool schemas into the Vercel AI SDK via the `tools` parameter, calling the sidecar's endpoints under the hood.

### Wiring MCP tools into Vercel AI SDK

```ts
// Fetch tool schemas from your Node sidecar
const toolSchemas = await fetch('http://localhost:3001/tools').then((r) => r.json())

// Convert to Vercel AI SDK tool format
const tools = Object.fromEntries(
  toolSchemas.map(({ name, description, inputSchema }) => [
    name,
    tool({
      description,
      inputSchema: jsonSchemaToZod(inputSchema), // use zod-from-json-schema
      execute: async (args) =>
        fetch(`http://localhost:3001/tools/${name}`, {
          method: 'POST',
          body: JSON.stringify(args),
        }).then((r) => r.json()),
    }),
  ])
)
```

## 10. Context Window Management

### The core problem

Local models have finite context windows (e.g., 8192 tokens for `llama3.2:3b`). Dumping full conversation histories causes:

- "Lost in the middle" degradation — models ignore content in long middles
- Silent truncation — Ollama clips silently when `num_ctx` is exceeded
- Slow inference — more tokens = more time per response

### Rolling summary pattern

```ts
interface ConversationMemory {
  sessionId: string
  summary: string // LLM-generated summary of older turns
  recentMessages: Message[] // last K turns kept verbatim
  tokenCount: number
}

async function buildContext(memory: ConversationMemory, newMessage: string) {
  const messages: Message[] = []

  // Inject summary as a system message
  if (memory.summary) {
    messages.push({
      role: 'system',
      content: `Conversation summary so far: ${memory.summary}`,
    })
  }

  // Add recent verbatim turns
  messages.push(...memory.recentMessages)
  messages.push({ role: 'user', content: newMessage })

  return messages
}

// Refresh summary every N turns
async function refreshSummary(memory: ConversationMemory) {
  const { text } = await generateText({
    model: ollama('llama3.2'),
    messages: [
      {
        role: 'user',
        content: `Summarize this conversation concisely in 3-5 sentences:\n\n${memory.recentMessages
          .map((m) => `${m.role}: ${m.content}`)
          .join('\n')}`,
      },
    ],
  })
  memory.summary = text
  memory.recentMessages = memory.recentMessages.slice(-4) // keep last 4
}
```

### Token budget tracking

Use Ollama's `eval_count` from the final stream chunk to track actual token consumption per response, and update your summary threshold dynamically.

## 12. Streaming + Thinking Mode UX

### Stream chunk shape

```ts
interface OllamaChunk {
  model: string
  created_at: string
  message: {
    role: 'assistant'
    content: string // accumulates the final response
    thinking?: string // accumulates reasoning (thinking models)
    tool_calls?: ToolCall[]
  }
  done: boolean
  // Only on final chunk when done: true:
  eval_count?: number
  total_duration?: number
}
```

### React component pattern

```tsx
function ChatMessage({ stream }: { stream: AsyncIterable<OllamaChunk> }) {
  const [thinking, setThinking] = useState('')
  const [content, setContent] = useState('')
  const [toolCalls, setToolCalls] = useState<ToolCall[]>([])
  const [stats, setStats] = useState<{ tokens: number; ms: number } | null>(null)

  useEffect(() => {
    ;(async () => {
      for await (const chunk of stream) {
        if (chunk.message.thinking) setThinking((t) => t + chunk.message.thinking)
        if (chunk.message.content) setContent((c) => c + chunk.message.content)
        if (chunk.message.tool_calls) setToolCalls(chunk.message.tool_calls)
        if (chunk.done && chunk.eval_count) {
          setStats({
            tokens: chunk.eval_count,
            ms: Math.round(chunk.total_duration! / 1e6),
          })
        }
      }
    })()
  }, [stream])

  return (
    <div>
      {thinking && (
        <details className='mb-2 text-sm text-gray-400'>
          <summary>Thinking…</summary>
          <p className='font-mono text-xs'>{thinking}</p>
        </details>
      )}
      <Markdown>{content}</Markdown>
      {toolCalls.map((tc) => (
        <ToolCallCard key={tc.function.name} call={tc} />
      ))}
      {stats && (
        <p className='text-xs text-gray-500'>
          {stats.tokens} tokens · {stats.ms}ms
        </p>
      )}
    </div>
  )
}
```

## 13. Capabilities & Shell Permissions

### `src-tauri/capabilities/default.json` (minimum required)

```json
{
  "$schema": "../gen/schemas/desktop-schema.json",
  "identifier": "default",
  "description": "Default capabilities for command center",
  "windows": ["main"],
  "permissions": [
    "core:default",
    "shell:allow-execute",
    "shell:allow-spawn",
    "shell:allow-kill",
    "fs:allow-read-text-file",
    "fs:allow-write-text-file",
    "fs:allow-read-dir",
    "notification:default",
    "os:allow-version"
  ]
}
```

### Security notes

- Never use `shell:allow-all` — it grants unrestricted process execution.
- Scope your `fs` permissions to `$APPDATA` and `$DOCUMENT` only; do not allow unrestricted root access.
- The `tauri-plugin-shellx` community plugin unlocks additional restrictions but should not be used in production.
